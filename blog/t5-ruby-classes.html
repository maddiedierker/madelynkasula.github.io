classes & methods, or

how to make your cat do stuff

20 july 2015

As usual, I'm going to talk to you about my cat (because she's awesome), but this time it will be via classes in Ruby. Take a look at the Cat class below:

We can see the cat will have name, gender, and color attributes. We used attr_reader because these are attributes that we do not want the class to be able to change--it can only view (or "read") them. We also need to initialize these variables in order to use them in our methods in the Cat class. Why do we need both, you ask? Using attr_reader allows other classes to access the values of these variables. So if I had a Dog class and the dog only chased cats named Spooky, I could create a new instance of the Cat class and call cat.name to determine whether this poor cat will be chased!

You'll notice that when I initialize my variables, I create instance variables as well. This allows all of the methods in my class to be able to access and do work on these variables. There is also a @lazy variable, which is not passed into the initialize method. That is because I do not need this information to be created with the new instance of my Cat class--I will work on it later. (I am also giving this cat the benefit of the doubt by initializing the variable to false, even though we know the cat is probably lazy.)

Now, we can create an instance of our Cat class, which is on line 34. The cat's name is Spooky, the cat's gender is female, and she is black and white in color (I prefer "tuxedo" as the phrase, but I suppose that isn't a color.) Again, because @lazy is not an argument that is passed into the initialize method, I only need to pass these three values in when I create a new cat. And that's how cats are made.

Let's make our cat do stuff! That's what our methods are for--forcing cats to do things they don't feel like doing. (Don't quote me on that.) You'll see we have three methods: speak, is_lazy?, and play. If, outside of our method (say, below line 34, since that is where we create cat), we were to call cat.[method_name_here], we could get the cat to say "Meow!", determine whether or not the cat is lazy by answering, "What does your cat do all day?", or get the cat to play with some string (if she is not lazy, of course.)

Let's look at the variable @lazy. First, in line 8, we initialize the variable to false. Then, in our is_lazy? method, we change the value of @lazy if the cat's owner's answer includes the words "sleeps" or "lays around." If it doesn't, @lazy is still false (re-stating this on line 21 is unnecessary, but makes it explicit). Lastly, in the play method, if the value was changed to true, then the cat will not play with you and will give you a sassy response (ugh, as usual). This passing of the @lazy variable between different methods throughout our class is why we need it to be an instance variable. Otherwise, we would have to explicitly pass the variable by returning its value in the is_lazy? method (which just so happens to be true in this case), creating a variable that equals the return value of is_lazy?, and passing that variable into the play method. Sometimes this is what we want because it creates more control for certain variables. However, we want our entire class to be able to access @lazy so that everyone knows how lazy this cat is.

And there we have it! Our Cat class models our cat and some of her many wonderful attributes and contains methods to make her hang out with me. Maybe I will just continue messing with my Cat class since all she does is sleep (which means she's lazy, which means she won't play with me).
